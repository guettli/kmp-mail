# KMP Mail Libraries — Implementation Plan

## Overview

Three Kotlin Multiplatform libraries for email protocol support:

| Library    | Protocols / RFCs                              | Dependencies                          |
|------------|-----------------------------------------------|---------------------------------------|
| `kmp-mime` | RFC 5322, RFC 2045, RFC 2047                  | Pure KMP — no network deps            |
| `kmp-smtp` | RFC 5321, AUTH PLAIN/LOGIN, STARTTLS          | ktor-network, ktor-network-tls        |
| `kmp-imap` | IMAP4rev1, IDLE, CONDSTORE, UIDPLUS           | ktor-network, ktor-network-tls        |

Dependency graph: `kmp-smtp` and `kmp-imap` both depend on `kmp-mime`.

## Developer Environment

### Nix Flake (`flake.nix`)

All required build tools are pinned in a Nix flake. Run once:

```sh
nix develop          # enter shell with JDK 21 + gradle + go-task
task init            # generate Gradle wrapper (./gradlew)
```

Tools provided:
| Tool       | Purpose                              |
|------------|--------------------------------------|
| JDK 21     | Kotlin compiler host + JVM tests     |
| gradle     | One-time `gradle wrapper` bootstrap  |
| go-task    | Taskfile runner                      |

After `task init`, `./gradlew` is the primary build tool; `gradle` from Nix is no longer needed.

## Test Fixture Strategy

### Decision

We do **not** copy JavaMail / Jakarta Mail APIs. Their tests are tightly coupled to
`javax.mail.*` Java types that cannot exist in `commonMain`. Mirroring their API would
produce a 1997-style blocking, properties-bag design that conflicts with ktor's coroutine model.

### What we take instead: raw protocol data

The valuable part of existing test suites is the **test data** — edge-case bytes that real
servers and libraries emit. We extract those as fixture files and load them with our own
test infrastructure:

| Module     | Fixture format | Source of data                                       |
|------------|----------------|------------------------------------------------------|
| `kmp-mime` | `.eml` files   | RFC 2045/2046/2047 appendix examples + JavaMail/Angus Mail test resources |
| `kmp-smtp` | `.txt` annotated transcripts | RFC 5321 §D examples, Postfix/Exim test logs |
| `kmp-imap` | `.txt` annotated transcripts | RFC 3501 §8 examples, Dovecot/Cyrus test logs |

### Fixture format — SMTP/IMAP transcripts

Lines prefixed `S:` are server-sent, `C:` are client-sent. Tests split on these
prefixes to feed the server side into a mock channel and assert the client side.

```
S: 220 mail.example.com ESMTP
C: EHLO client.example.com
S: 250-mail.example.com
S: 250-STARTTLS
S: 250 OK
```

### Fixture loader helper (`commonTest`)

Each module gets a small `Fixtures.kt` in `commonTest` that reads files from
`resources/fixtures/<module>/`. On JVM this uses `ClassLoader.getResourceAsStream`;
the `expect`/`actual` pattern covers Native and JS.

### Existing open-source fixture sources

| Project          | URL (for manual extraction)                                     |
|------------------|-----------------------------------------------------------------|
| Angus Mail       | `https://github.com/eclipse-ee4j/angus-mail` — `mail/src/test/`|
| Apache James MIME4J | `https://github.com/apache/james-mime4j` — `core/src/test/`|
| Dovecot test suite | `https://github.com/dovecot/core` — `src/tests/`            |

Extract raw `.eml` or protocol log snippets only; do not copy any Java test code.

## Module: `kmp-mime`

### Key Design Decisions

- **No external dependencies** — only `kotlinx-coroutines-core` for `Flow`-based streaming.
- The parser is **incremental**: it accepts `ByteArray` chunks so it can be driven by a ktor channel.
- Header values are stored raw; structured access (e.g., `ContentType`, `Address`) is provided by separate value types that parse on demand.
- `TransferEncoding` codec uses `expect`/`actual` where base64 delegates to platform (JVM: `java.util.Base64`, JS: `atob`/`btoa`, Native: pure-KMP impl).

### Public API Sketch

```kotlin
// Parse
val message: MimeMessage = MimeParser.parse(rawBytes)

// Build
val raw: ByteArray = MimeBuilder {
    from("Alice <alice@example.com>")
    to("Bob <bob@example.com>")
    subject("Hello")
    textBody("Hello, World!")
}.build()

// Encoded-word
val decoded: String = HeaderEncoding.decode("=?UTF-8?B?SGVsbG8=?=")
val encoded: String = HeaderEncoding.encodeIfNeeded("Héllo")
```

### RFC Coverage

| RFC    | Feature                        | Scope |
|--------|--------------------------------|-------|
| 5322   | Message format, folding, dates | Full  |
| 2045   | MIME headers, body encoding    | Full  |
| 2046   | Multipart, message/rfc822      | Full  |
| 2047   | Encoded-word (B and Q)         | Full  |
| 2183   | Content-Disposition            | Full  |

## Module: `kmp-smtp`

### Public API Sketch

```kotlin
val client = SmtpClient {
    host = "smtp.example.com"
    port = 587
    security = SmtpSecurity.STARTTLS
    credentials = SmtpCredentials("user@example.com", "password")
}

client.connect()
client.sendMessage(mimeMessage)
client.disconnect()
```

### Session State Machine

```
GREETING → EHLO → [STARTTLS → TLS handshake → EHLO again]
        → AUTH (PLAIN or LOGIN) → READY
        → MAIL FROM → RCPT TO (×n) → DATA → BODY → . → 250
        → QUIT
```

### AUTH Mechanisms

- `PLAIN`: `\0username\0password` base64-encoded in one step.
- `LOGIN`: username/password in two separate base64 challenge/response pairs.

### Error Handling

- All 4xx / 5xx responses raise `SmtpException(code, message)`.
- `STARTTLS` failure is fatal — never fall back to plain text.

## Module: `kmp-imap`

### Public API Sketch

```kotlin
val client = ImapClient {
    host = "imap.example.com"
    port = 993
    security = ImapSecurity.TLS
    credentials = ImapCredentials("user@example.com", "password")
}

client.connect()
val mailbox = client.select("INBOX")
val uids = mailbox.search(ImapQuery.unseen())
val messages = mailbox.fetchByUid(uids, FetchProfile.fullHeaders())

// IDLE
mailbox.idle { event ->
    when (event) {
        is ImapEvent.Exists  -> println("New mail: ${event.count} messages")
        is ImapEvent.Expunge -> println("Deleted: seqno ${event.seqno}")
    }
}

client.disconnect()
```

### Session State Machine (RFC 3501 §3)

```
Not Authenticated → LOGIN / AUTHENTICATE → Authenticated
Authenticated     → SELECT / EXAMINE      → Selected
Selected          → CLOSE / UNSELECT      → Authenticated
Any               → LOGOUT               → Logout
```

### Extension Support

| Extension  | RFC   | Commands / Capabilities                                         |
|------------|-------|-----------------------------------------------------------------|
| IDLE       | 2177  | `IDLE`, `DONE`                                                  |
| CONDSTORE  | 7162  | `MODSEQ` in `FETCH`/`SEARCH`, `HIGHESTMODSEQ` in `SELECT` OK   |
| UIDPLUS    | 4315  | `APPENDUID`, `COPYUID` in tagged OK responses                   |

### Response Parser

IMAP responses are complex (nested parenthesised lists, literals, atoms). The parser handles:

- Untagged: `* n EXISTS`, `* FETCH (...)`, `* FLAGS (...)`, `* CAPABILITY ...`
- Tagged: `A001 OK`, `A001 NO`, `A001 BAD`
- Continuation: `+ go ahead`
- Literals: `{42}\r\n<42 bytes>`
- Quoted strings and atom lists

## Open Questions / Constraints

1. **JS target**: `ktor-network` on JS requires Node.js (not browser). Browser builds must stub network modules.
2. **Android**: Libraries target JVM; Android consumers add AGP in their own project.
3. **LITERAL+**: Non-synchronising literals (RFC 7888) supported if advertised in `CAPABILITY`.
4. **Thread safety**: All clients designed for single-coroutine use; external sync required for concurrent access.
5. **OAuth2 / XOAUTH2**: Out of scope for v1; AUTH mechanism interface is open for extension.
